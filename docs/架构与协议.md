| 字段 | 值 |
|------|-----|
| 状态 | 工程规格 |

## 1. 目标与非目标

### 1.1 目标

1. Desktop 作为 **指挥面 / ACP Client（+ 本地 Host）**，复用现有 agent runtime。
2. 会话 **格式**与 CLI/agent 兼容；Desktop 数据目录默认 **`~/.grok-desktop`**，与 CLI `~/.grok` 隔离（避免互相覆盖）。
3. 提供产品级能力所需的 **Roster、Attach、Worktree、Change feed、Goal/Subagent 事件、Automations、Inbox** 等稳定 API。
4. 单机多窗口、托盘长驻、安全默认、可观测、可升级。

### 1.2 非目标

1. 在 Renderer 或无关服务中重写 tools / sampling / MCP 运行时。
2. 发明与 ACP 平行的第二套「聊天协议」承载 turn 语义（允许 Host 旁路 API，但不复制 agent 智能）。
3. 用 `grok -p` headless 轮询冒充实时多 agent 指挥台。

│  · Project/Thread 编目与投影                                    │
│  · Agent 进程生命周期（spawn / attach / restart）               │
│  · Automations 调度                                             │
│  · Inbox 聚合                                                   │
│  · 与 leader / 文件系统 / git 协作                              │

                │ ACP JSON-RPC/stdio 等    │ 文件系统
┌───────────────▼──────────────┐  ┌────────▼─────────────────────┐
│ Grok Agent Runtime           │  │ 数据平面（Desktop）            │

**依赖方向：** Presentation → Host → Runtime / FS。禁止 Runtime 依赖 Desktop UI。

## 3. 进程与部署模型

### 3.1 推荐进程拓扑

| 进程 | 职责 | 生命周期 |
|------|------|----------|
| **Shell Main** | Electron/Tauri 主进程、窗口 | 随 UI |
| **Host** | 单实例编排（可与 Main 同进程，逻辑上独立模块） | 应用级；托盘可保持 |
| **Agent Worker(s)** | `grok` 子进程，承载 session | 按 Thread 策略 |
| **可选 Leader** | 现有 leader 多客户端共享 | 若采用 leader 附着模式 |

### 3.2 单实例

- 全机（或每用户） **一个 Host**。
- 第二次启动：聚焦已有窗口 / 打开 deep link，不写双份 session。
- 实现：OS 单实例锁 + 本地 IPC（命名管道 / socket / 锁文件），路径纳入 `~/.grok-desktop/desktop/`。

### 3.3 Thread ↔ 进程策略

按实现成熟度分两档，**产品语义相同**：

| 模式 | 说明 | 适用 |
|------|------|------|
| **A. Leader 多 session** | Host 连接 leader，多 Thread 多 session 共进程池 | 资源优；需 leader API 完整 |
| **B. 每 Thread 一 stdio agent** | `grok agent stdio`（或等价）每 Thread 一子进程 | 隔离强；内存高 |

**要求：**

- 同一 Thread 同时只允许 **一个可写附着者**（Desktop 窗口或 CLI attach 策略明确）。
- 非附着窗口可只读订阅投影状态（roster + 落盘 transcript）。
- 进程崩溃：Host 标记 `failed`，可 restart+resume；不丢磁盘 session。

### 3.3.1 Mode B 附着状态机（P1-B）

Desktop 默认 **懒附着**：打开会话只读历史（`history_only`），不 spawn agent。发送、预热「连接 Agent」、或需 ACP 的 slash 时进入 `attaching` → 成功为 `live`。

```
openThread → history_only
用户发送 / 连接 Agent / 需 ACP 操作
         → attaching → live
                         │ idle TTL（P3-C）
                         ▼
                    detaching → history_only（保留 sessionId / 磁盘 transcript）
live + crash/ping fail → failed → reattach → attaching
```

| 状态 | 含义 | UI |
|------|------|-----|
| `history_only` | 无可写 agent 进程 | pill「仅历史」；可点连接 |
| `attaching` | spawn/load 中 | pill「正在连接…」 |
| `live` | stdio 可写 | pill「已连接」 |
| `failed` | ping/进程失败 | pill + 崩溃条 + 重连 |
| `detaching` | idle 回收中 | 短暂 |

**Host API：** `threads.attach` / `threads.detach` / `threads.ping` / `threads.attachState`。  
**Renderer：** `ensureLiveThread` 在 live 时先 ping，失败不假 live。

### 3.3.2 Prompt 队列 L1 / L2（P1-C · P2-A）

| 层 | 真相源 | 路径 / 协议 |
|----|--------|-------------|
| **L1 本地权威** | Host 落盘 | `~/.grok-desktop/desktop/queues/{sessionId}.json` |
| **L2 Agent 同步** | 可选 | 附着后 `QueueChanged` + `x.ai/queue/*`；失败保留 L1 并提示「仅本地队列」 |

- 切会话 / 重启应用：按 `sessionId` 重载 L1，**不**静默清空。  
- interrupt 后 `pausedByInterrupt`：不自动 drain，用户 Resume 后按序发送。  
- Host IPC：`queue.get|set|enqueue|remove|reorder|update|clear|setFlags`。

### 3.3.3 Idle detach（P3-C）

Mode B 下多会话各占一 agent 进程。Host 周期扫描：

- 超过 `idleDetachMs` 且非 working / 无 pending permission / 无后台 task → detach，状态回 `history_only`。  
- 再发送时 `ensureLiveThread` 重新 attach，历史不丢。  
- 设置可关闭（`idleDetachMs: 0`）或调整上限 LRU。

实现：`src/host/attach-policy.ts` + Host 定时循环。

### 3.4 二进制解析顺序

与实现 `resolve-grok.ts` 一致：

1. 用户 Settings / `GROK_DESKTOP_AGENT`（override）
2. 应用捆绑：`agent-bin/`（开发）或 `resources/agent/`（安装包）
3. `PATH` / `~/.grok/bin` / `~/.grok-desktop/bin`

启动时记录实际路径与 `grok --version`；若同目录有 `VERSION.txt`（sync:agent 生成），关于页展示 version / sha256 / 同步时间。

### 3.5 壳层技术

| 选项 | 建议 |
|------|------|
| **Electron + TS** | 默认推荐：对齐 Codex 工程路径，Diff/终端/富文本生态成熟 |
| **Tauri 2** | 若包体/内存为硬约束；**Host API 与产品规格不变** |

下文 IPC 以「Main ↔ Renderer typed bridge」描述，不绑定具体框架 API 名。

## 4. 数据平面与 `~/.grok-desktop` 契约

### 4.1 原则

- Desktop 将 **`GROK_HOME` 设为 `~/.grok-desktop`**，与 CLI 默认 `~/.grok` **目录隔离**（避免互相覆盖 auth / sessions）。
- **Session 文件格式**仍与 agent/CLI 兼容；Host 内存是缓存，真相源在磁盘。
- Desktop **不得**另起一套 session JSON schema 与 CLI 分叉。
- 仅 Desktop 的索引（pin、归档、Automation、Inbox 已读）放在 `~/.grok-desktop/desktop/`，且能映射回 `session_id` / project path。
- 旧版若曾写入 `~/.grok/desktop/`，启动时可迁移到 `~/.grok-desktop/desktop/`（见 `paths.ts`）。

### 4.2 目录约定（逻辑）

> 具体文件名以运行时实现为准；下列为 Host 必须对接的逻辑区域。

| 区域 | 用途 |
|------|------|
| `~/.grok-desktop/config.toml` | Desktop 用户配置 |
| `~/.grok-desktop/sessions/…` | 会话与 `plan.md` 等 |
| `~/.grok-desktop/plugins/`、skills 发现路径 | 扩展 |
| 项目 `.grok/`、`.grok/plugins/` 等 | 项目级配置与插件 |
| `~/.grok-desktop/desktop/lock` | 单实例 |
| `~/.grok-desktop/desktop/projects.json` | Project 注册表（路径、pin、别名） |
| `~/.grok-desktop/desktop/automations/` | Automation 定义与 run 记录 |
| `~/.grok-desktop/desktop/inbox/` | Inbox 持久化（或 sqlite） |
| `~/.grok-desktop/desktop/logs/` | Host/UI 日志 |
| `~/.grok/bin`（只读参考） | 本机 CLI 二进制候选，Desktop 不写入 |

### 4.3 Project 注册

```jsonc
// ~/.grok-desktop/desktop/projects.json（示意）
{
  "version": 1,
  "projects": [
    {
      "id": "proj_…",
      "path": "D:/work/foo",
      "title": "foo",
      "pinned": true,
      "archived": false,
      "trust": "trusted",
      "created_at": "…",
      "last_opened_at": "…"
    }
  ]
}
```

- `path` 规范化（Windows 大小写/分隔符、解析 junction 策略写明）。
- 删除 Project 默认 **不删除** git worktree 与 sessions；可提供高级清理。

### 4.4 与 CLI 互通

| 操作 | 期望 |
|------|------|
| CLI 新建 session | 进入 Host roster（watch 目录或 leader 事件） |
| Desktop 新建 Thread | 磁盘 session 可被 `grok` resume |
| 改 `config.toml` | 双边最终一致（Host 监听或显式 reload） |
| Skills/Plugins 安装 | 共用发现路径 |

## 5. 运行时集成：ACP

### 5.1 基线

现有入口（见 `xai-grok-shell` README）：

```bash
grok agent stdio
```

- 传输：JSON-RPC over stdio（JSONL）
- 协议：[Agent Client Protocol](https://agentclientprotocol.com)
- 库：TS `@agentclientprotocol/sdk`；Rust `agent-client-protocol` / 本仓 `xai-acp-lib`

Desktop Host 作为 **ACP Client**：

- 实现 Client 侧能力：`request_permission`、读写文件/终端相关回调（按 runtime 需求）、`session_notification` 消费
- 发起 Agent 侧方法：`session/new`、`session/load`、`session/prompt` 等（以实际 ACP 与 Grok 扩展为准）

### 5.2 与 Codex App Server 的对应关系

| Codex | Grok |
|-------|------|
| App Server binary | `grok` agent runtime |
| JSON-RPC JSONL stdio | ACP JSON-RPC stdio |
| Desktop 捆绑 server | Desktop 捆绑 `grok` |
| 多 surface 共用 server | CLI / IDE / Desktop 共用 runtime |

### 5.3 ACP 扩展策略

1. **标准 ACP 优先** 承载 turn、工具流、权限。
2. **Grok 专有字段** 走 `_meta` 或文档化的 method 扩展，版本化。
3. **指挥台级查询**（全机 roster、跨 session diff 聚合）可由 **Host API** 提供，内部组合：leader 协议 + 扫 session 目录 + git + 可选 agent 扩展方法。
4. 扩展必须 **向后兼容**：旧 Desktop 连新 `grok` 可降级；新 Desktop 连旧 `grok` 隐藏不可用功能。

### 5.4 推荐：Session 事件归一化

Host 将 ACP `session_notification` 与扩展事件归一为 UI 事件流（示意）：

| 归一事件 | 来源 | UI 用途 |
|----------|------|---------|
| `turn.started` / `turn.completed` | ACP | 状态点、时间线 |
| `message.delta` | ACP | Transcript 流式 |
| `thought.delta` | ACP/扩展 | 思考折叠 |
| `tool.started` / `tool.completed` | ACP | 工具卡片 |
| `permission.requested` | Client 回调 | 卡片 + Inbox |
| `question.requested` | 扩展/工具 | 结构化问答 |
| `plan.updated` | 扩展/FS watch | Plan 面板 |
| `goal.updated` | 扩展 | Goal Rail |
| `subagent.updated` | 扩展 | Agent Tree |
| `change.updated` | hunk/git 聚合 | Changes |
| `session.status` | roster | Command Center |

UI **只订阅归一事件**，不直接解析多种底层格式。

## 6. Desktop Host API（目标契约）

> 下列为 **目标态 Host RPC**。实现语言不限（TS 模块或 Rust sidecar）。  
> 方法名示意；落地时用 OpenAPI/JSON Schema 或 `ts-rs`/prost 生成类型。

### 6.1 约定

- 传输：Main↔Host 同进程函数调用，或 `Host ↔ Renderer` 经 IPC。
- 所有写操作可失败；返回结构化 `Error { code, message, details? }`。
- 列表接口支持 `cursor` 分页（session 历史可能很大）。
- 订阅：`subscribe(topic) -> unsubscribe`；事件至少 once 送达，允许重复（UI 幂等）。

## 7. Desktop IPC（Shell ↔ Renderer）

### 7.1 原则

- **无裸 `any` 总线**：共享 schema 包（`src/shared`）。
- Renderer **不直接 spawn** `grok`、不直接写 session 目录（开发模式可开旁路旗标）。
- 大负载（diff、历史）走流式或分页，避免一次 IPC 塞数 MB。
- 权限决策只在 Main/Host 信任域最终执行。

### 7.2 通道类型

| 通道 | 用途 |
|------|------|
| `invoke` 请求/响应 | CRUD、prompt、respond |
| `event` 推送 | roster、turn 事件、inbox |

## 8. Leader 协议关系

现有 `xai-grok-shell` leader（length-prefixed JSON frames）用于多 pager 共享。

**Host 可选集成：**

1. **优先 attach leader**（若用户本机已有 leader）以共享 session 进程。  
2. 无 leader 时 Host **自起** agent/leader。  
3. Desktop 不重新发明 frame 协议；封装在 Host 适配层。

注意：leader 消息与 ACP 是不同层；UI 只看 Host 归一模型。

## 9. 安全架构

### 9.1 信任边界

```
[Renderer] 不可信（XSS 面）
    ↓ 严格 IPC
[Main/Host] 半信任（本地用户）
    ↓ spawn + ACP
[Agent Runtime] 工具执行 + sandbox
    ↓
[OS / 网络 / 文件系统]
```

### 9.2 硬性要求

1. Renderer 加载内容 CSP：禁止随意远程脚本（除更新/遥测白名单）。
2. Markdown 渲染消毒；`javascript:` 链接禁用。
3. `openInEditor` / `openPath` 校验路径，防 `../../../` 逃逸到非用户意图位置（可提示）。
4. Automation `runNow` 与定时触发走同一权限配置文件。
5. 密钥仅存现有 auth 存储；UI 只显示掩码。
6. 日志默认脱敏（见 `xai-grok-secrets` 方向）。

### 9.3 权限响应路径

```
Agent tool 需批准
  → ACP request_permission
  → Host 生成 PermissionRequest + InboxItem + OS 通知（可选）
  → UI 卡片
  → permissions.respond
  → 回到 ACP Client 回调结果
```

超时：可配置；超时默认 **deny**（安全）并 Inbox 标记。

### 9.4 Sandbox

- 复用 `xai-grok-sandbox` 与 shell 权限模式，不在 Desktop 再实现内核沙箱。
- Settings 只暴露产品层档位，映射到现有 config 键。

## 16. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 多进程内存膨胀 | leader 模式、空闲 Thread 休眠、并发上限 |

| Automation 狂建 worktree | 清理策略、pin 限制、磁盘预算警告 |
| XSS via Markdown | 消毒组件、CSP、禁用危险协议 |
